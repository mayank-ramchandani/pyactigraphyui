# Deployment and operations

## Build identity

Set:

```text
APP_VERSION=<release-or-image-tag>
GIT_COMMIT=<commit-sha>
VITE_APP_VERSION=<frontend-release>
```

Confirm the deployed backend using `GET /api/version` after every release.

## Frontend environment

```text
VITE_API_BASE_URL=https://your-backend.example/
VITE_GITHUB_REPOSITORY_URL=https://github.com/owner/repository
VITE_ENABLE_AUTH_RUNS=false
```

The repository URL is optional. When provided, the in-app Documentation page exposes a direct link to `docs/` on GitHub.

## CORS

Add frontend origins through:

```text
CORS_ALLOW_ORIGINS=https://frontend-one.example,https://frontend-two.example
```

Do not solve a CORS problem by allowing untrusted origins in production.

## Persistent application data

Set `APP_DATA_DIR` to a mounted persistent path when feedback, diagnostics, and progress state must survive container revisions:

```text
APP_DATA_DIR=/data/actigraphy-ui
```

Without persistent storage, `/tmp` data is ephemeral.

## Large uploads

Every layer must accept the intended file size:

- browser/frontend hosting path;
- Nginx or ingress;
- Azure Container Apps configuration;
- any upstream gateway;
- application server and timeout;
- temporary storage.

An HTTP 413 generated by Nginx is not fixed by changing FastAPI alone.

## Memory and concurrency

A raw file can expand substantially during decoding. Prefer streamed processing, avoid loading high-frequency XYZ into a complete Pandas DataFrame, and account for concurrent users.

Review:

- container memory limit;
- worker count;
- Java heap when converter paths are used;
- raw chunk/page size;
- temporary output size;
- simultaneous analyses.

GT3X streaming controls:

```text
GT3X_STREAM_CHUNK_SECONDS=300
GT3X_PROGRESS_EVENT_INTERVAL=100000
GT3X_DEDUPE_WINDOW_SECONDS=300
GT3X_ISM_FILL_CHUNK_SECONDS=300
```

The defaults keep raw GT3X buffers small. Increasing the chunk duration can
improve throughput slightly but also increases peak memory. Multiple workers or
simultaneous large requests multiply the per-request memory requirement.

Large-file preview and conversion endpoints are synchronous FastAPI handlers so
their CPU work runs in the server thread pool rather than blocking the event
loop. Uploaded temporary files are deleted in `finally` blocks on success and
failure.

## Request duration

Streaming fixes the raw-decoding memory amplification but does not remove
gateway request deadlines. Measure the complete browser upload, GT3X reduction,
and selected analysis time. If their sum approaches the platform deadline, use
a background-job endpoint and polling instead of extending application timeouts
alone.

## Progress polling with replicas

Progress is request-scoped and may be mirrored to `APP_DATA_DIR`. For multiple replicas, use shared storage or sticky routing. Otherwise, the poll can reach a replica that does not hold the request state.

## Diagnostic settings

```text
DIAGNOSTIC_LOG_MAX_MB=50
DIAGNOSTIC_SHA256_MAX_MB=512
DIAGNOSTIC_TRACEBACK_CHARS=12000
DIAGNOSTIC_SUPPRESSED_ERROR_LIMIT=30
GENEACTIV_DIAGNOSTIC_PAGE_INTERVAL=5000
GT3X_PROGRESS_EVENT_INTERVAL=100000
ANALYSIS_PROGRESS_TTL_SECONDS=21600
EXPOSE_SERVER_ERROR_DETAILS=false
```

Expose detailed server errors only during controlled debugging.

## Release procedure

1. Merge and tag the intended commit.
2. Build frontend and backend from the same commit.
3. Deploy the backend image.
4. Confirm `/api/version` and feature flags.
5. Deploy the frontend with the correct backend URL.
6. Run a small golden file.
7. Run a large-file diagnostic test.
8. Download and inspect diagnostics.
9. Update `docs/CHANGELOG.md`.
